# Harness 工程实战（三）：上下文管理——最容易被忽视的瓶颈

---

大多数人在评估 AI 系统时，注意力都集中在模型能力上。模型能做什么？它有多聪明？

但实际在生产环境中跑过复杂 Agent 系统的人都知道，真正的瓶颈往往不在模型，而在上下文。

你花了大价钱请了顶级专家，结果他一进门就忘了你三句话之前说的话。这不是他不聪明，是你没给他一个好的工作环境。

上下文管理，就是给模型建立那个工作环境的技术。

## Lost in the Middle

斯坦福有一项经典研究叫"Lost in the Middle"。

当关键信息落在上下文窗口的中间位置时，模型性能下降 30% 以上。Chroma 后来做了更深入的实验，发现同样的问题在他们自己的系统中复现。

这不是模型的问题。这是人类设计的上下文结构没有考虑到模型 attention 机制的天然缺陷。

模型的 attention 不是均匀分布在整个上下文上的。开头和结尾的信息更容易被注意到，中间的信息容易"迷失"。

![上下文窗口的 Lost in the Middle 问题](./images/harness-context-window.jpg)

一个百万 token 的上下文窗口听起来很大，但如果设计不当，实际有效的信息密度可能远低于预期。

## 五种生产策略

**第一种：基于时间的清除。**

最简单粗暴的策略。当对话超过一定时间或轮次，直接清除早期内容，保留最近的对话。

优点是实现简单。缺点是可能误删重要上下文。如果第 3 轮提到了关键设计约束，到第 40 轮时这条信息可能早就被清除了。

**第二种：对话总结。**

更智能的做法：当上下文接近限制时，对历史对话进行压缩总结，保留"大意"而非细节。

Claude Code 在这方面做得很细致：保留架构决策和未解决的 bug，丢弃冗余的工具输出。工具调用产生的原始数据量很大，但真正有价值的是人类可读的总结。

**第三种：观察遮蔽。**

JetBrains 的 Junie 采用了另一种思路：隐藏旧的工具输出，但保持工具调用本身可见。

这背后的洞察是：模型需要知道"做过什么"，但不一定需要"怎么做的"具体细节。就像你看简历，想知道候选人做过哪些项目，不需要知道他每天几点下班。

**第四种：即时检索。**

维护轻量级的标识符，而不是加载完整数据。动态查询，按需获取。

Claude Code 使用 grep、glob、head、tail 而不是加载完整文件。这看起来是微优化，但在处理大型代码库时，这决定了上下文窗口的消耗速度。

```python
# 不是这样
with open("large_codebase.py") as f:
    content = f.read()  # 全部加载，上下文爆炸

# 而是这样
result = subprocess.run(
    ["grep", "-n", "class Agent", "large_codebase.py"],
    capture_output=True, text=True
)  # 只加载匹配结果，上下文高效
```

**第五种：子智能体委托。**

每个子智能体广泛探索，但只返回 1000 到 2000 token 的压缩摘要。

这个策略解决的是：如果让一个 Agent 去探索一个很大知识库，全量返回所有细节会把主上下文撑爆。分而治之：子 Agent 做深度探索，主 Agent 只接收摘要。

## ACON 的发现

2025 年的 ACON 研究提供了一个重要的定量结论：

通过优先考虑推理轨迹而不是原始工具输出，token 减少 26% 到 54%，同时保持 95% 以上的准确性。

这个发现很反直觉。按照直觉，会觉得保留越多的工具输出越好。但实际上，模型的推理过程比原始数据更有价值。

这就像面试一个人，你更想知道他是怎么思考问题的，而不是他背出来的那些事实。

## 上下文工程的目标

Anthropic 的上下文工程指南说了一句话，是整个领域的核心原则：

找到最小的高信号 token 集合，最大化期望结果的可能性。

不是上下文越多越好。上下文太多，噪声淹没信号，模型反而更容易迷失。真正的优化方向是：让模型看到的每一条信息都是它此刻真正需要的。

好的上下文管理不是塞入更多信息，是剔除无关信息。

## 实际工程中的陷阱

有一个常见的陷阱是"过度准备上下文"。

工程师的直觉是准备越充分，模型表现越好。所以把能想到的所有上下文都塞进去：项目背景、代码结构、历史讨论、技术选型理由……

结果上下文窗口爆炸，模型在大量无关信息中迷失，真正重要的设计约束反而落在窗口中部被忽略了。

另一个陷阱是"上下文状态不一致"。

Agent 的记忆说项目用的 PostgreSQL，但实际代码里配的是 MySQL。这种不一致会导致模型基于错误信息做决策。

Claude Code 的设计原则是：Agent 把自己的记忆当作"提示"，在行动前会对照实际状态验证。记忆可能过时，必须校验。
