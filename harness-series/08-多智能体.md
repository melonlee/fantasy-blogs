# Harness 工程实战（八）：多智能体系统的工程实践——何时拆、怎么拆、如何通信

---

大多数时候，单个 Agent 够用。

Anthropic 和 OpenAI 都说过同样的话：首先最大化单个 Agent。

这句话重要到值得重复：先让单个 Agent 做他能做的一切，只有在明确需要时才拆成多个。

因为多 Agent 系统有显著的开销。

## 拆分的多重代价

多 Agent 不是"一个 Agent 做不到的，两个 Agent 加起来就能做到"。

每次从一个 Agent 交接给另一个，你都在损失上下文。两个 Agent 需要通信，需要额外的 LLM 调用来做路由，需要管理状态同步。

这些开销加起来可能超过拆分带来的收益。

LLMCompiler 报告比顺序 ReAct 快 3.6 倍，靠的是把规划和执行分离，避免每一步都等待模型思考。但这种加速的前提是规划的开销比分步执行的总和要小。

多 Agent 也是同样的道理：只有在任务足够复杂，拆分后的收益超过通信开销时，拆分才有意义。

## 什么时候应该拆

两个明确的触发条件。

**第一个：工具过载。**

当一个 Agent 需要使用大约超过 10 个重叠工具时，模型开始混淆。工具越多，"我应该用哪个"这个问题就越消耗模型的推理能力。

这种情况下，按功能域拆分是自然的。一个 Agent 处理文件操作，另一个处理网络请求，第三个处理代码分析。每个 Agent 只需要知道自己的工具集，不需要知道其他 Agent 能做什么。

**第二个：明显独立的任务域。**

如果两个子任务需要完全不同的专业知识、完全不同的工具集、完全不同的上下文，那拆分几乎是必然的。

比如，一个 Agent 负责代码生成，另一个 Agent 负责代码审查。前者需要代码库的上下文，后者需要安全和质量的标准。这两个任务的上下文重叠很少，放在一起反而增加复杂度。

## 三种通信模式

Claude Code 支持三种子 Agent 执行模型，对应三种不同的通信模式。

**Fork：父上下文的字节相同副本。**

子 Agent 拿到和父 Agent 完全相同的上下文，然后独立探索。

这个模式的代价是：上下文在拆分时一次性复制。如果父 Agent 有大量上下文，这个复制成本很高。

收益是：子 Agent 完全独立，可以做任何事情，不受父 Agent 的约束。

**Teammate：基于文件的邮箱通信。**

子 Agent 运行在独立的终端窗格，通过文件进行通信。父 Agent 写消息到文件，子 Agent 读取并回复。

这个模式引入了异步通信的可能。子 Agent 可以花很长时间处理一个复杂任务，父 Agent 不需要等待。

代价是通信效率低。文件读写比内存共享慢得多，而且没有结构化的通信协议。

**Worktree：独立的 git 工作树。**

每个 Agent 有自己的 git 分支，可以独立提交代码。

这是最重的模式，但提供了最强的隔离性。不同 Agent 的修改不会相互干扰，最后可以通过 git merge 合并。

适合场景：多个 Agent 需要并行处理同一个代码库的不同部分，而且每个部分都可能产生冲突的修改。

## 交接协议

OpenAI 的 SDK 支持两种多 Agent 交互模式。

**Agent 作为工具。**

专家 Agent 被当作工具调用，用完即弃。父 Agent 决定什么时候调用哪个专家，然后把任务完全委托出去。

这适合任务边界清晰、子任务结果独立的场景。比如，一个 Agent 专门做代码搜索，另一个专门做代码生成。父 Agent 决定搜什么，然后决定用哪个结果来生成。

**交接。**

父 Agent 把完全控制权交给专家 Agent，自己等待结果。

这适合需要专家深入处理、中间结果可能影响最终输出的场景。比如，用户说"帮我改进这个函数"，代码审查 Agent 发现了问题，需要把这个发现传递给代码修改 Agent。

交接比"Agent 作为工具"更重，但更灵活。

## 嵌套状态图

LangGraph 将子 Agent 实现为嵌套状态图。

每个子 Agent 是父状态图中的一个节点，这个节点内部有自己的状态流转逻辑。

这提供了最精确的控制粒度。你可以为子 Agent 定义独立的节点和边，独立的状态管理和验证逻辑。

代价是复杂性。嵌套层数变多后，状态流转的追踪变得困难。

一个实用的原则是：嵌套层数不要超过三层。

超过三层后，状态管理的复杂度会超过分层带来的结构清晰度。

## 多 Agent 的验证

单 Agent 的验证已经够复杂。多 Agent 的验证是另一个级别。

每个 Agent 可能产生错误，Agent 之间的交接可能丢失信息，整个系统的端到端正确性需要从每个 Agent 的正确性和它们之间的交互来推导。

一个实用的策略是：验证每个 Agent 的局部正确性，加上端到端的行为验证。

局部正确性用单 Agent 的验证方法：测试、截图、LLM 评判。

端到端行为验证看的是整体效果：最终输出是否满足用户意图？中间过程是否有明显的信息丢失？

分层验证比试图在每一步都做完整验证更scalable。

## 什么时候多 Agent 是过度设计

最后一个问题：什么时候应该选择多 Agent，但实际不需要？

一个信号是：你在设计多 Agent 协作流程时，脑子里想象的交互图比实际的计算图复杂得多。

如果设计出来的系统比你用单 Agent 实现的等效功能还复杂，说明拆分变成了过度设计。

另一个信号是：你花了更多时间在设计 Agent 之间的通信协议，而不是在解决实际问题。

通信协议很重要，但如果它变成了项目的主要工作内容，说明架构选择可能有问题。

多 Agent 是手段，不是目的。最终关心的是任务完成的质量和效率，不是系统架构的"优雅程度"。
