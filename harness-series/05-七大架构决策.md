# Harness 工程实战（五）：七大架构决策——每个 Harness 设计者都必须回答的问题

---

设计一个 Harness 不是选技术栈。技术选型有文档可查，有社区经验可参考。

真正的挑战是那些没有标准答案的问题：什么时候该拆 Agent？什么时候该加验证？Harness 应该厚还是薄？

这些问题背后是七个核心的架构决策。每个决策都需要根据具体场景来判断。

## 决策一：单 Agent 还是多 Agent

这是最常被问到的决策。

Anthropic 和 OpenAI 都给了一个相同的建议：首先最大化单个 Agent。

多 Agent 系统看起来更强大，但会增加显著的开销：路由需要额外的 LLM 调用，Agent 之间的交接会丢失上下文，管理多个 Agent 的状态比管理一个复杂得多。

什么时候应该拆分？

两个触发条件：
- 工具过载：大约超过 10 个重叠的工具，模型开始混淆
- 明显独立的任务域：不同领域的任务需要完全不同的工具集和专业技能

拆分不是银弹。如果单个 Agent 能做好，拆分成多 Agent 只会增加复杂度。

## 决策二：ReAct 还是计划-执行

ReAct 在每一步交织推理和行动。模型思考"我应该做什么"，然后执行，再思考，再执行。

优点是灵活——模型可以根据上一步的结果调整下一步的计划。

缺点是每一步都有成本。思考本身需要 token，执行也需要 token。

计划-执行将规划与执行分离。模型先花时间制定完整计划，然后一次性执行。

LLMCompiler 报告比顺序 ReAct 快 3.6 倍。

这是一个很大的差距。背后的原因是：计划制定后，具体执行步骤可以并行，不需要每一步都等待模型思考。

选择取决于任务类型：
- 探索性任务，目标是发现而非执行 → ReAct
- 执行路径清晰的任务 → 计划-执行

## 决策三：上下文窗口管理策略

五种生产方法各有权衡：

| 策略 | 优点 | 缺点 |
|------|------|------|
| 基于时间的清除 | 简单 | 可能丢失重要上下文 |
| 对话总结 | 保留大意 | 细节可能丢失 |
| 观察遮蔽 | 减少噪声 | 需要实现遮蔽逻辑 |
| 即时检索 | 最小化上下文 | 增加延迟 |
| 子 Agent 委托 | 处理复杂任务 | 需要设计摘要协议 |

没有完美的策略。只有适合当前场景的策略。

## 决策四：验证循环设计

第二章已经详细讨论过验证循环。这里关注的是架构层面的选择：

**计算验证**（测试、linter）提供确定性真相。测试通过就是通过，失败就是失败。但只能验证能用规则定义的内容。

**推理验证**（LLM 作为评判者）可以捕获语义问题。输出是否满足用户意图？这个很难用测试定义，但 LLM 可以判断。

代价是：计算验证几乎零成本，推理验证有显著延迟和费用。

一个好的架构设计通常两者都有。计算验证先行，拦住所有能用规则判断的错误。推理验证在后，处理需要语义判断的质量问题。

## 决策五：权限和安全架构

这是最容易在开发阶段被忽视的决策。

设计系统时，工程师通常假设一切正常。但生产环境中会有各种异常：工具调用超时、API 返回异常值、网络分区……

权限架构回答的问题是：Agent 在面对未知情况时，默认行为是什么？

两种选择：

**宽松模型**：快速行动，自动批准大多数操作。优点是流畅，缺点是有安全风险。

**限制模型**：每个操作都需要确认。优点是安全，缺点是流程缓慢。

选择取决于部署环境。一个内部工具可能适合宽松模型。一个处理敏感数据的系统需要限制模型。

Claude Code 管理约 40 个离散工具能力，分三个阶段建立信任：项目加载时、每次工具调用前、高风险操作时。这个渐进式的信任建立值得参考。

## 决策六：工具范围策略

更多工具通常意味着更差的性能。

这听起来违反直觉。直觉上，更多工具意味着 Agent 能力更强。

但实际上，工具越多，模型需要决定的上下文就越复杂。"我应该用哪个工具？"这个问题本身就开始消耗模型的推理能力。

Vercel 从 v0 中删除了 80% 的工具，获得了更好的结果。

Claude Code 通过延迟加载实现了 95% 的上下文减少。模型在某个步骤只需要知道当前可用的工具，不需要知道所有可能的工具。

原则是：暴露当前步骤所需的最小工具集，而不是所有工具全集。

## 决策七：Harness 的厚度

这是最根本的架构押注。

**薄 Harness 方案**：模型是智能的来源，Harness 只做最少的事情。编排循环、工具调用、状态管理——所有"机械"的事情由 Harness 做，所有"思考"由模型做。

Anthropic 押注于薄 Harness。他们相信随着模型变强，Harness 会自然变薄。现在需要的一些逻辑，在更强的模型里可能是多余的。

**厚 Harness 方案**：Harness 包含更多逻辑。规划步骤、策略选择、错误恢复——这些不是交给模型，而是显式写在 Harness 里。

基于图的框架（LangGraph）押注于显式控制。他们认为有些事情 Harness 应该决定，不应该全部委托给模型。

两种方案都有成功的案例，也都有失败的情况。

## 这些决策之间的关系

这七个决策不是独立的。

单 Agent vs 多 Agent 影响工具范围策略的选择。ReAct vs 计划-执行 影响验证循环的设计。权限架构影响 Harness 的厚度选择。

这些决策会相互强化或相互矛盾。

好的架构设计不是单独优化每个决策，而是找到一组相互一致的整体决策。

一个薄 Harness 系统不太适合复杂的权限架构，因为很多事情是 Harness 决定不了的。一个多 Agent 系统如果用厚 Harness，每个 Agent 之间的协调逻辑会变得非常复杂。

在做这些决策时，想清楚它们之间的依赖关系，比单独研究每个问题更重要。
