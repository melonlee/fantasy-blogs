# Harness 工程实战（三）：上下文管理——最容易被忽视的瓶颈

---

大多数人在评估一个 AI 系统时，注意力都集中在模型能力上：模型能做什么？它有多聪明？

但实际在生产环境中跑过复杂 Agent 系统的人都知道，真正的瓶颈往往不在模型，而在上下文。

你花了大价钱请了顶级专家，结果他一进门就忘了你三句话之前说的话。这不是他不聪明，是你没给他一个好的工作环境。

上下文管理，就是给模型建立那个工作环境的技术。

## Lost in the Middle

斯坦福有一项经典研究叫"Lost in the Middle"。

这项研究测试的是：当关键信息落在上下文窗口的中间位置时，模型的表现会怎么样？

结果很残酷：关键内容落在窗口中部时，模型性能下降 30% 以上。

Chroma 后来做了更深入的实验，发现同样的问题在他们自己的系统中复现。

这不是模型的问题。这是人类设计的上下文结构没有考虑到模型 attention 机制的天然缺陷。

模型的 attention 不是均匀分布在整个上下文上的。开头和结尾的信息更容易被注意到，中间的信息容易"迷失"。

一个百万 token 的上下文窗口，听起来很大，但如果设计不当，实际有效的信息密度可能远低于你的预期。

## 五种生产策略

经过大量实践，业界总结出了五种主要的上下文管理策略。

**第一种：基于时间的清除。**

最简单粗暴的策略。当对话超过一定时间或轮次，直接清除早期内容，保留最近的对话。

优点是实现简单。缺点是可能误删重要上下文。如果一个 50 轮的对话中，第 3 轮提到了关键的设计约束，到第 40 轮时这条信息可能早就被清除了。

**第二种：对话总结。**

更智能一点的做法：当上下文接近限制时，对历史对话进行压缩总结，保留"大意"而非细节。

Claude Code 在这方面做得很细致：保留架构决策和未解决的 bug，丢弃冗余的工具输出。工具调用产生的原始数据量很大，但真正有价值的是人类可读的总结。

**第三种：观察遮蔽。**

JetBrains 的 Junie 采用了另一种思路：隐藏旧的工具输出，但保持工具调用本身可见。

这背后的洞察是：模型需要知道"做过什么"，但不一定需要"怎么做的"具体细节。就像你看一个人的简历，你想知道他做过哪些项目，但不需要知道他每天几点下班。

**第四种：即时检索。**

维护轻量级的标识符，而不是加载完整数据。动态查询，按需获取。

Claude Code 使用 grep、glob、head、tail 而不是加载完整文件。这看起来是微优化，但在处理大型代码库时，这决定了上下文窗口的消耗速度。

这是一个很重要的工程原则：能用搜索解决的，不要用加载。

**第五种：子智能体委托。**

每个子智能体广泛探索，但只返回 1000 到 2000 token 的压缩摘要。

这个策略解决的是一个问题：如果你让一个 Agent 去探索一个很大知识库，全量返回所有细节会把你的主上下文撑爆。

解决方法是分而治之：子 Agent 做深度探索，主 Agent 只接收摘要。

## ACON 的发现

2025 年的 ACON 研究提供了一个重要的定量结论：

通过优先考虑推理轨迹而不是原始工具输出，token 减少 26% 到 54%，同时保持 95% 以上的准确性。

这个发现很反直觉。

按照直觉，你会觉得保留越多的工具输出越好——越多的原始数据，模型应该判断越准确。

但实际上，模型的推理过程（它是怎么想的）比原始数据（它看到了什么）更有价值。

这就像面试一个人，你更想知道他是怎么思考问题的，而不是他背出来的那些事实。

## 上下文工程的目标

Anthropic 的上下文工程指南说了一句话，我觉得是整个领域的核心原则：

找到最小的高信号 token 集合，最大化期望结果的可能性。

这不是说上下文越多越好。上下文太多，噪声淹没信号，模型反而更容易迷失。

真正的优化方向是：让模型看到的每一条信息都是它此刻真正需要的。

好的上下文管理不是塞入更多信息，是剔除无关信息。

## 实际工程中的陷阱

有一个常见的陷阱是"过度准备上下文"。

工程师的直觉是，准备越充分，模型表现越好。所以把能想到的所有上下文都塞进去：项目背景、代码结构、历史讨论、技术选型理由……

结果呢？上下文窗口爆炸，模型在大量无关信息中迷失，真正重要的设计约束反而落在窗口中部被忽略了。

另一个陷阱是"上下文状态不一致"。

Agent 的记忆说项目用的 PostgreSQL，但实际代码里配的是 MySQL。这种不一致会导致模型基于错误信息做决策。

Claude Code 的设计原则是：Agent 把自己的记忆当作"提示"，在行动前会对照实际状态验证。这很重要——记忆可能过时，必须校验。

## 三层记忆架构

Claude Code 实现了一个三层层次结构，这个设计值得详细说：

**第一层：轻量级索引。** 每条约 150 字符，始终加载。这是模型在任何时候都能快速访问的概要信息。

**第二层：按需拉取的详细文件。** 当模型需要某个文件的详细内容时，主动加载，用完可以释放。

**第三层：仅通过搜索访问的原始记录。** 完整的历史记录，但只通过搜索获取，不主动加载到上下文。

这个设计的核心思想是：分层管理，按需使用。

不是所有信息都需要一直放在上下文里。有些信息只需要能被找到，不需要一直"在场"。

## 上下文管理和模型能力的关系

有一个趋势值得注意：随着模型能力的提升，上下文管理的策略也在演变。

早期的模型上下文窗口小，必须精细管理。2025 年的模型已经有了百万 token 级别的上下文，但管理策略不是变简单了，而是变得更复杂了——因为窗口大了之后，更容易陷入"我到底该放什么进去"的困境。

上下文管理的本质不是窗口大小的问题，是信息优先级的问题。

不管窗口多大，这个问题都存在。
