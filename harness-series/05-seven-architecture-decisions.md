# Harness 工程实战（五)：七大架构决策——每个 Harness 设计者都必须回答的问题

---

设计一个 Harness 不是选技术栈。真正的挑战是那些没有标准答案的问题：什么时候该拆 Agent？什么时候该加验证？Harness 应该厚还是薄？

这些问题背后是七个核心的架构决策。每个决策都需要根据具体场景来判断。

![架构决策的权衡](./images/harness-architecture-decisions.jpg)

## 决策一：单 Agent 还是多 Agent

Anthropic 和 OpenAI 都给了相同的建议：首先最大化单个 Agent。

多 Agent 系统看起来更强大，但会增加显著的开销：路由需要额外的 LLM 调用，Agent 之间的交接会丢失上下文，管理多个 Agent 的状态比管理一个复杂得多。

什么时候应该拆分？两个触发条件：
- 工具过载：大约超过 10 个重叠的工具，模型开始混淆
- 明显独立的任务域：不同领域的任务需要完全不同的工具集

## 决策二：ReAct 还是计划-执行

ReAct 在每一步交织推理和行动。优点是灵活，可以根据上一步的结果调整下一步的计划。缺点是每一步都有成本。

计划-执行将规划与执行分离。模型先花时间制定完整计划，然后一次性执行。

LLMCompiler 报告比顺序 ReAct 快 3.6 倍。原因是：计划制定后，具体执行步骤可以并行，不需要每一步都等待模型思考。

```python
# ReAct 模式
while not done:
    thought = model.think(messages)
    action = model.decide_action(thought)
    result = execute(action)
    messages.append(result)

# 计划-执行模式
plan = model.plan(task)  # 一次性生成完整计划
for step in plan.steps:
    result = execute(step)
    verify(step, result)
```

## 决策三：上下文窗口管理策略

| 策略 | 优点 | 缺点 |
|------|------|------|
| 基于时间的清除 | 简单 | 可能丢失重要上下文 |
| 对话总结 | 保留大意 | 细节可能丢失 |
| 观察遮蔽 | 减少噪声 | 需要实现遮蔽逻辑 |
| 即时检索 | 最小化上下文 | 增加延迟 |
| 子 Agent 委托 | 处理复杂任务 | 需要设计摘要协议 |

## 决策四：验证循环设计

**计算验证**（测试、linter）提供确定性真相，但只能验证能用规则定义的内容。

**推理验证**（LLM 作为评判者）可以捕获语义问题，但有显著延迟和费用。

一个好的架构设计通常两者都有。计算验证先行，拦住所有能用规则判断的错误。推理验证在后，处理需要语义判断的质量问题。

## 决策五：权限和安全架构

两种选择：

**宽松模型**：快速行动，自动批准大多数操作。优点是流畅，缺点是有安全风险。

**限制模型**：每个操作都需要确认。优点是安全，缺点是流程缓慢。

Claude Code 分三个阶段建立信任：项目加载时、每次工具调用前、高风险操作时。

## 决策六：工具范围策略

更多工具通常意味着更差的性能。

Vercel 从 v0 中删除了 80% 的工具，获得了更好的结果。原则是：暴露当前步骤所需的最小工具集。

## 决策七：Harness 的厚度

**薄 Harness** 押注于模型变强。Anthropic 押注于这个方向。

**厚 Harness** 押注于显式控制。基于图的框架（LangGraph）押注于此。

两种方案都有成功的案例，也都有失败的情况。

## 这些决策之间的关系

这七个决策不是独立的。单 Agent vs 多 Agent 影响工具范围策略的选择。权限架构影响 Harness 的厚度选择。好的架构设计不是单独优化每个决策，而是找到一组相互一致的整体决策。
