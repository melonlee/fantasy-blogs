# Harness 工程实战（六）：主流框架横评——Anthropic SDK vs OpenAI Agents SDK vs LangGraph vs CrewAI

---

工具好不好用，不能只看功能列表。

Claude Code 在 Codex 界面上比在通用聊天窗口中表现更好。这不是因为模型不同，是因为 Harness 不同。

这说明一个框架的核心价值不在于它"能做什么"，在于它"怎么做"——它的设计哲学、它的默认假设、它让你以什么方式思考问题。

当前 Agent 框架生态里，有四个值得认真研究的系统。

## Anthropic Claude Agent SDK

Anthropic 的 SDK 是"哑循环"哲学的最佳体现。

核心抽象是一个 `query()` 函数：创建 Agent 循环，返回流式消息的异步迭代器。就这么多。

循环本身是"哑"的，所有智能都在模型里。Harness 不做推理，不做规划，只做机械的事情：组装提示词、调用模型、解析输出、执行工具、返回结果。

这个设计选择背后有一个信念：模型会越来越强。让模型做推理比让 Harness 做推理更 scalable。如果某个逻辑今天需要写在 Harness 里，明天模型升级后可能就不需要了。

Claude Code 的实现走得更远。它用收集-行动-验证循环：收集上下文（搜索文件、读代码）、采取行动（编辑文件、运行命令）、验证结果（运行测试、检查输出）。

状态管理的选择也很有意思：用 git 提交作为检查点，进度文件作为结构化草稿本。

这让它天然具备版本控制能力。每次重要的状态变更都有 git commit，可以随时回滚。

## OpenAI Agents SDK

OpenAI 的 SDK 是"代码优先"的设计。

工作流逻辑用原生 Python 表达，而不是图 DSL。你写的是 Python 代码，不是某种框架特定的描述语言。

这降低了学习曲线。任何 Python 开发者都能理解 Agents SDK 的代码，不需要学习一套新的框架概念。

通过 `Runner` 类实现 Harness，有三种模式：异步、同步和流式。

工具支持三种类型：
- 函数工具（`@function_tool` 装饰器）
- 托管工具（WebSearch、CodeInterpreter、FileSearch）
- MCP 服务器工具

多 Agent 支持两种模式：
- **Agent 作为工具**：专家处理有界子任务，用完即弃
- **交接**：专家获得完全控制，原 Agent 等待结果

状态管理提供四种互斥策略：应用程序内存、SDK 会话、服务器端 Conversations API，或轻量级 `previous_response_id` 链接。

这是一个务实的设计。不同场景需要不同的状态管理，SDK 不强迫你选一种。

## LangGraph

LangGraph 将 Harness 建模为显式状态图。

这不是图 DSL——图是 Python 代码写的。你定义节点（Python 函数）和边（条件判断），LangGraph 帮你处理执行和状态流转。

两个核心节点：`llm_call` 和 `tool_node`，通过条件边连接。如果存在工具调用，路由到 `tool_node`；如果不存在，路由到 END。

这个模型的优势是控制粒度。复杂的流程逻辑可以精确表达，包括条件分支、循环、并行执行。

状态建模为流经图节点的类型化字典，用 reducer 合并更新。检查点发生在超级步骤边界，支持中断后恢复和时间旅行调试。

LangGraph 从 LangChain 的 AgentExecutor 演变而来。AgentExecutor 在 v0.2 中被弃用，原因是难以扩展且缺乏多 Agent 支持。

LangGraph 是对 AgentExecutor 局限的回应。

## CrewAI

CrewAI 实现基于角色的多 Agent 架构。

三个核心概念：
- **Agent**：围绕 LLM 的 Harness，由角色、目标、背景故事和工具定义
- **Task**：工作单元
- **Crew**：Agent 集合

CrewAI 的独特之处是"角色"概念。每个 Agent 被赋予一个人格：背景故事、行为模式、专业领域。这让多 Agent 协作更接近人类团队的工作方式。

Flows 层添加了"具有智能的确定性骨干"。这处理路由和验证，而 Crews 处理自主协作。

区分这两件事很重要：确定性逻辑（流程控制、路由规则）应该显式管理，不应该委托给 Agent 的"自行判断"。

## 横评对比

| 维度 | Claude Agent SDK | OpenAI Agents SDK | LangGraph | CrewAI |
|------|------------------|-------------------|-----------|--------|
| 核心抽象 | query() 函数 | Runner 类 | 状态图 | 角色 + Crew |
| 多 Agent | Fork/Teammate/Worktree | 交接 + Agent 作为工具 | 嵌套状态图 | Role-based Crews |
| 状态管理 | git + 进度文件 | 四种策略选择 | 检查点 + 时间旅行 | 任务状态 |
| 验证 | 内置验证循环 | 输入/输出/工具三级 | 可扩展 | 可扩展 |
| 学习曲线 | 低 | 低 | 中 | 中 |
| 适用场景 | 深度编码任务 | 通用 Agent | 复杂流程编排 | 多角色协作 |

## 关键洞察：没有银弹

看完这四个框架，一个重要的事实是：没有哪个框架在所有场景都优于其他。

Claude Agent SDK 的"哑循环"哲学在编码场景表现出色，但在需要复杂流程控制的场景显得能力不足。

OpenAI Agents SDK 的"代码优先"降低了门槛，但复杂流程的代码可能变得难以维护。

LangGraph 的状态图提供了最强表达力，但学习成本和调试成本也最高。

CrewAI 的角色抽象让多 Agent 协作很自然，但如果你不需要角色扮演，这个抽象就多余。

选择框架时，最重要的是想清楚你的场景需要什么。

一个需要深度定制、流程复杂的系统，LangGraph 的表达能力值得投入学习成本。一个相对简单的 Agent 应用，OpenAI Agents SDK 的代码优先设计能快速出活。

不要因为某个框架"最流行"就选它。
