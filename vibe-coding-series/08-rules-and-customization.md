# Vibe Coding 实战（八）：Rules 与定制——让 AI 理解你的项目规范

---

用 Cursor 或 Claude Code 这类工具写代码的时候，你有没有遇到过这种情况：AI 生成了一段代码，你一看，方向是对的，但命名风格不对、注释格式不对、或者一些团队里约定俗成的做法它完全没遵守。然后你就要花时间改，改完之后再让 AI 继续。这个来回拉锯的过程很磨人。

问题出在上下文。你知道项目的规范是什么，AI 不知道。你在这个项目里泡了几个月甚至几年，踩过坑、定过规矩、形成过默契，这些信息 AI 通通没有。每次你让 AI 写代码，它都是在"信息不完整"的状态下做决策，输出自然会有偏差。

Rules 就是来解决这个问题的。你可以把项目的编码规范、技术选型、架构约束、团队约定，以文本配置文件的形式告诉 AI，让它每次生成代码的时候都把这些规范考虑进去。这样 AI 不再是在真空中生成代码，而是在"了解你家情况"的前提下来帮你。

## Cursor Rules 是什么

Cursor 的 Rules 功能本质上是一个持久化的上下文注入机制。当你给 Cursor 写一条 Rule，这条规则会被附加到每一个 AI 请求的 system prompt 里，无论你是在 Composer 模式聊天、在 Editor 模式补全代码、还是用 Cmd+K 问问题，AI 都能读到这条规则。

Rules 的格式是 Markdown 文本，你可以在里面写任何想让 AI 知道的信息。规范、约定、示例代码、项目背景——都可以。我见过有人用 Rules 来做这几件事：定义代码风格（缩进、命名、分号）、指定技术栈版本（React 18、TypeScript 5）、说明项目结构（哪些目录放什么）、描述 API 设计原则（RESTful 规范）、甚至记录"我们团队不这么写"的反模式。

Cursor 的 Rules 分两种：全局 Rules 和项目级 Rules。全局 Rules 存放在 `~/.cursor/rules/` 目录下，对所有项目生效。项目级 Rules 放在项目根目录的 `.cursor/rules/` 目录下，只在当前项目里生效。两套规则可以同时存在，Cursor 会把全局 Rules 和项目级 Rules 合并后一起注入。如果有冲突，项目级 Rules 优先级更高。

## Rules 的分层：全局 vs 项目级

我建议大部分规则放在项目级，全局只放那些"无论写什么项目都适用"的基础规则。这样做的好处是避免规则过于臃肿——你肯定不想每次写代码都让 AI 读完一堆跟当前项目无关的规范。

全局 Rules 里适合放什么？编程语言的基础语法规范、通用代码风格（比如用 2 空格缩进、变量名用 camelCase）、常见的性能陷阱规避（比如不要在循环里查数据库）、安全底线（不许在前端代码里硬编码密钥）。这些规范不依赖具体项目，任何代码仓库都应该遵守。

项目级 Rules 要写什么？第一是项目结构和技术栈——用的是什么框架、什么数据库、目录怎么组织。第二是业务约定——哪些模块负责什么、模块之间的依赖关系是怎样的。第三是代码风格的具体要求——比如你们前端用 ESLint + Prettier 强制统一格式，或者后端用某套命名规范。这些信息只有项目里的人知道，AI 只有读取了这些上下文才能生成符合要求的代码。

Rules 的分层还要考虑一个问题：什么时候从全局降级到项目级。假设你在做一个开源项目，代码风格想保持统一，但每个具体项目有自己的技术栈和规范。这种情况下，全局 Rules 放通用的开源贡献指南（怎么写 commit message、PR 要包含哪些信息），项目级 Rules 放具体技术实现。两种 Rules 各司其职，不打架。

## 怎么写有效的 Rules

Rules 写得好不好，直接决定了 AI 的输出质量。我见过两种典型的失败案例。一种是 Rules 写得太多太碎，把 AI 的 context window 塞满了，AI 在读取规则上花的 token 比实际生成代码的 token 还多，效率反而下降。另一种是 Rules 写得太空泛，全是"要写高质量代码"这种正确的废话，AI 读完之后该怎样还是怎样。

有效的 Rules 有几个共同特征。

**具体，不抽象**。与其说"代码要清晰可读"，不如说"每个函数不超过 40 行，超过 40 行必须拆分"。与其说"错误处理要做好"，不如说"所有外部 API 调用必须 try-catch，catch 里要记录具体错误信息并给出用户友好的提示"。具体的规则能被 AI 理解和执行，抽象的规则 AI 读完就忘了。

**有正例，也有反例**。告诉 AI"要这样写"的同时，告诉它"不要那样写"。比如你想说明函数命名规范，可以写一个正面例子：`getUserById(id: string)` ，再写一个反面例子：`getUser(id)` ——后者缺少类型信息，容易产生歧义。AI 对比着看，比纯文字描述理解得更准确。

**控制长度，聚焦关键**。Rules 不是项目百科全书，不需要把所有细节都写进去。只写 AI 容易出错的地方、团队有特殊要求的地方、新人容易踩坑的地方。常规的代码风格让工具（ESLint、Prettier）去强制，Rules 只管工具覆盖不到的部分。

说一个我自己的反面案例。我之前给一个前端项目写 Rules，洋洋洒洒写了三千多字，把 React 的每一条最佳实践都列了进去。结果 AI 生成代码的时候，要么因为规则太长而忽略部分内容，要么生成出来的代码过于教条，每个组件都按照规则模板套一遍，灵活性和可读性反而下降了。后来我大幅精简，只保留了 10 条最核心的规则，AI 的输出质量反而提高了。

**反面案例： Rules 写得空泛**

```markdown
# 代码风格规范

代码要写得清晰、专业、高效。遵循最佳实践，
避免常见的代码异味。注意代码的可读性和可维护性。
```

这条规则写了等于没写，AI 读完之后不知道具体该做什么。任何代码都可以说自己"清晰、专业、高效"。

**正面案例：具体到可执行**

```markdown
# 组件规范

1. 组件文件放在 `src/components/` 目录下，按功能模块再分子目录
2. 组件必须是函数组件，禁止使用 class 组件
3. Props 必须定义 interface，文件名格式：`ComponentNameProps.ts`
4. 组件内禁止直接使用 `any` 类型，所有类型必须显式声明
5. 条件渲染使用 `{condition && <Component />}` 而不是三元运算符 `condition ? <Component /> : null`
```

这样的规则 AI 能直接理解和执行，输出质量自然有保障。

## Rules 的维护和演进

Rules 不是写完就完事的，需要随着项目一起演进。新人加入团队会提出新的规范想法，项目技术栈升级会淘汰一些旧规则，业务变化会带来新的约束。这些都需要及时反映到 Rules 里。

我维护 Rules 的方式是：每次 Code Review 发现一个 AI 反复犯的同类错误，就把这个错误的规避方法加到 Rules 里。比如 AI 每次生成数据库操作的代码都忘记关闭连接，我就在 Rules 里加一条"所有数据库操作必须在 finally 块中关闭连接"。几次积累下来，Rules 变得越来越厚道，AI 犯错的频率也越来越低。

Rules 的版本管理也值得关注。把 Rules 文件放进 Git 一起管理，每次修改都走 PR 流程，让团队成员一起 review 新规则是否合理。这样做的好处是 Rules 的变更有记录可查，不会出现"某天 Rules 被改掉了但没人知道"的情况。

还有一个实践是定期 review Rules 本身。每个季度花半小时过一遍 Rules，看哪些规则已经过时、哪些规则从来没人触发过、哪些新场景需要补充。我之前 review 的时候发现，有两条规则是两年前加的，但对应的那段代码已经被删掉了——这两条规则就属于冗余，应该清理掉。

Rules 的积累本质上是团队知识沉淀的过程。好的 Rules 让新加入的开发者能快速理解项目的编码规范，也让 AI 能像一个"了解项目背景的老成员"一样工作。随着 Rules 越来越完善，你和 AI 的协作效率会越来越高，摩擦会越来越少。

## 用 Rules 解决实际问题

说几个 Rules 帮我解决的实际问题。

第一个是 API 响应格式的统一。我们项目里所有 API 响应都有固定格式：`{ code, message, data }`，但 AI 每次生成的 API 端点都是"裸"返回，格式各异。写了几条 Rules 规定响应格式之后，AI 生成的前后端接口代码就再没出过这个问题。

第二个是敏感信息的处理。AI 有时候会无意间在代码里留下密钥或者内网地址的痕迹，这在安全审计的时候会变成问题。我加了一条 Rules 规定"所有密钥必须从环境变量读取，禁止硬编码"，并且给了几个正面和反面示例。之后 AI 生成代码的时候就会主动避免这个问题。

第三个是测试代码的规范。AI 生成的测试代码覆盖率还行，但格式乱七八糟——有的用 describe-it 结构，有的用 test 函数，有的甚至混在一起用。统一了测试规范之后，代码审查的时候看测试代码就清爽多了。

Rules 不是万能的，它不能替代你对代码质量的判断。但它能大幅减少 AI 在"基础规范"上消耗的摩擦，让你把精力放在真正需要思考的地方。

## Rules 与其他机制的配合

Rules 只是上下文管理的一种方式，它需要和其他机制配合才能发挥最大效果。

最基础的配合是 Rules + Linter/Formatter。Rules 告诉你"要怎么做"，Linter/Formatter 强制你"必须这样做"。AI 生成代码可能不完全遵守 Rules，但最终产出的代码会经过 Linter 检查，不合规的会被标出来。这个双重保障很重要。

另一个配合是 Rules + 项目文档。Rules 写的是编码规范，但项目的业务逻辑、架构设计、技术选型理由，这些信息放在 README 或者 ARCHITECTURE.md 里更合适。Rules 里可以引用这些文档，比如"详细的技术选型理由见 ARCHITECTURE.md"，AI 需要的时候会去找对应文档读。这样 Rules 就不用写得过长，信息也不重复。

还有 Rules + Commit Hook。AI 生成的代码提交之前，用 commit hook 跑一遍测试和 lint，失败的不能提交。这个机制帮你建立了一个安全网——无论 AI 生成的代码质量如何，最终上线的代码都经过了基本的质量检查。

## 开始写你的第一条 Rule

如果你现在还没用过 Rules，建议从今天开始给自己定一个规则：每次发现 AI 犯了一个"不该犯"的错误，就把规避方法写成一条 Rule 加进去。积累几个月之后，你会发现 AI 的输出质量在稳步提升，而你花在"纠正 AI"上的时间在逐步减少。

这个过程本身也是对自己项目规范的重新审视。很多时候，你以为自己知道项目的规范是什么，但当你试图把这些规范写成 AI 能理解的文字时，才会发现原来有些约定是模糊的、没有形成文字的。把它写出来的过程，也是把隐性知识显性化的过程，对自己对团队都有好处。
