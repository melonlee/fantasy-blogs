# Vibe Coding 实战（十）：全流程演练——用 vibe coding 开发一个完整功能

---

前面几篇文章讲了很多方法论和技巧，这篇换个方式——我们从头到尾走一遍完整的 vibe coding 开发流程，看一个具体的功能是怎么从需求变成上线的。

我选了一个比较典型的场景：给一个博客系统添加"文章收藏"功能。这个功能涉及前端交互、后端接口、数据库设计、状态管理，是一个完整的端到端功能，开发过程中会遇到各种决策点，很适合用来演示 vibe coding 的实际工作方式。

![完整工作流的卡通图：电脑屏幕显示从需求到部署的完整流程，周围有忙碌的小机器人](./images/10-full-process.jpg)

## 需求描述

在开始写代码之前，先把需求说清楚。

博客系统是一个简单的 CMS，用户可以发表文章、阅读文章。现在需要加一个收藏功能：已登录用户可以收藏自己喜欢的文章，已收藏的文章在个人中心可以查看，文章详情页需要显示当前用户是否已收藏。

技术栈：前端 React + TypeScript，后端 Node.js + Express，数据库 PostgreSQL。前端状态管理用 React Query，后端没有使用 ORM，直接写 SQL。

## 第一步：写 Spec

这是 vibe coding 里最重要的一步，也是大多数人最容易跳过的步骤。我见过太多人直接给 AI 说"帮我加一个收藏功能"，然后期待 AI 把所有事情都做好。AI 不是你肚子里的蛔虫，它需要足够清晰的需求描述才能生成正确的结果。

我写的 Spec 是这样的：

```
# 文章收藏功能 Spec

## 功能需求

1. 登录用户可以在文章详情页收藏/取消收藏文章
2. 文章详情页显示当前用户对该文章的收藏状态（已收藏/未收藏）
3. 个人中心有"我的收藏"页面，展示用户收藏的所有文章列表
4. 收藏操作需要登录，未登录用户点击收藏时跳转登录页

## 接口设计

### 收藏文章
POST /api/articles/:id/favorite
- 需要认证
- 返回 200 + { favorited: true }
- 文章不存在返回 404
- 未登录返回 401

### 取消收藏
DELETE /api/articles/:id/favorite
- 需要认证
- 返回 200 + { favorited: false }
- 文章不存在返回 404
- 未登录返回 401

### 获取收藏状态
GET /api/articles/:id/favorite
- 需要认证
- 返回 200 + { favorited: boolean }
- 文章不存在返回 404

### 获取用户收藏列表
GET /api/users/me/favorites
- 需要认证
- 返回 200 + { favorites: Article[] }，按收藏时间倒序
- 分页：page, pageSize 参数，默认 page=1, pageSize=20

## 数据模型

### favorites 表
- id: 主键自增
- user_id: 引用 users(id)，非空
- article_id: 引用 articles(id)，非空
- created_at: 收藏时间
- 复合唯一索引：(user_id, article_id)，防止重复收藏

## 前端交互

1. 文章详情页：收藏按钮显示在文章底部，图标按钮（空心星/实心星），点击切换状态
2. 收藏状态在页面加载时从 GET /api/articles/:id/favorite 获取
3. 按钮点击后乐观更新 UI，同时发送请求，后端返回后修正状态
4. 收藏操作有 3 秒防抖，避免用户快速点击产生多余请求
5. "我的收藏"页面在个人中心 tab 里，路由 /profile/favorites

## 错误处理

- 网络错误：显示 toast 提示"操作失败，请重试"，不改变 UI 状态
- 401 错误：跳转登录页，登录成功后继续操作
- 404 错误：显示 toast 提示"文章不存在"

## 技术约束

- 前端使用 React Query 管理收藏状态
- 收藏状态在数据库层面保证一致性，不在应用层做重复检查
- 所有接口需要防 SQL 注入
```

这个 Spec 写得很详细，包含了功能需求、接口设计、数据模型、交互细节、错误处理和技术约束。把这个 Spec 发给 AI，它能理解要做什么、做到什么程度、不能做什么。

## 第二步：数据库迁移

拿到 Spec 之后，我先让 AI 生成数据库迁移脚本。数据库是基础设施，需要先搭好，后面的接口和前端都依赖它。

```
请基于以下数据模型生成 PostgreSQL 迁移脚本：

创建 favorites 表：
- id: 主键自增
- user_id: integer, 引用 users(id)，非空
- article_id: integer, 引用 articles(id)，非空
- created_at: timestamp，默认当前时间
- 复合唯一索引：(user_id, article_id)

使用原生 SQL，不要使用 ORM。
```

AI 生成的迁移脚本：

```sql
-- 创建 favorites 表
CREATE TABLE IF NOT EXISTS favorites (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    article_id INTEGER NOT NULL REFERENCES articles(id) ON DELETE CASCADE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, article_id)
);

-- 创建索引优化查询性能
CREATE INDEX idx_favorites_user_id ON favorites(user_id);
CREATE INDEX idx_favorites_article_id ON favorites(article_id);
```

看起来没问题。我运行了迁移，确认表创建成功。

## 第三步：后端接口

数据库就绪之后，开始写后端接口。我一个一个来，不是一次性让 AI 生成所有接口——一次生成太多，AI 容易顾此失彼，出错了排查也困难。

先写收藏接口（POST /api/articles/:id/favorite）：

```
请实现 POST /api/articles/:id/favorite 接口

要求：
- 需要认证（通过 req.user.id 获取当前用户 ID）
- 先检查文章是否存在，不存在返回 404
- 插入 favorites 表记录
- 如果已经收藏过（UNIQUE 约束冲突），返回已收藏状态，不报错
- 返回 { favorited: true }
- 防 SQL 注入（使用参数化查询）
```

AI 生成的代码我 review 了一遍，整体 OK，但有一个问题：它没有处理 UNIQUE 约束冲突的情况，当用户重复收藏时会抛 500 错误而不是正常返回。我把这个问题反馈给 AI：

```
当前实现有问题：如果用户已经收藏过，再调用这个接口会因为 UNIQUE 约束冲突抛出 500 错误。
需要修改：捕获唯一约束冲突错误，返回 { favorited: true }，不抛异常。
```

AI 修复了这个 bug。

取消收藏接口（DELETE /api/articles/:id/favorite）也是类似的流程，先让 AI 生成，review 发现问题再反馈修正。

获取收藏状态和收藏列表的实现相对顺利，没有遇到大的问题。

这里有一个 vibe coding 的技巧：当 AI 生成完一段代码之后，不要直接接受，而是先问自己"这段代码在什么情况下会出错"。想清楚之后，跑几个边界 case 验证一下，确认没问题再继续。

## 第四步：前端实现

后端接口写完并测试通过之后，开始做前端。

先写文章详情页的收藏按钮。这个组件的核心逻辑是：加载时获取收藏状态，用户点击时切换状态，显示对应的图标。

我给 AI 的 prompt：

```
请在文章详情页添加收藏功能：

组件需求：
- 收藏按钮：星形图标（空心=未收藏，实心=已收藏）
- 按钮放在文章底部、作者信息之后
- 点击按钮切换收藏状态
- 加载中状态显示旋转图标，按钮不可点击
- 未登录用户点击时，跳转 /login?redirect=当前页面路径

状态管理：
- 使用 React Query 的 useQuery 和 useMutation
- 乐观更新：点击时立即更新 UI，后端返回后回滚（如失败）
- 3 秒防抖

UI 参考：
- 使用 Heroicons 的 star 图标
- 颜色：已收藏 gold，未收藏 gray
```

AI 生成完代码，我跑了一下，有两个问题。一是图标尺寸不对，显示太小；二是乐观更新逻辑写反了——它把失败回滚写成了"成功时回滚"，我赶紧指出来让它改。

```
有两处问题：
1. 图标 size 属性应该用 "24"，目前是 "20"，显示太小
2. 乐观更新的回滚逻辑写反了：onMutate 更新缓存，onError 应该回滚到之前的状态，而不是再更新一次
```

AI 修复了这两个问题。

## 第五步：个人中心收藏列表

这个页面展示用户收藏的文章列表，需要分页。

```
请实现 /profile/favorites 页面：

功能需求：
- 展示当前用户收藏的所有文章
- 按收藏时间倒序排列
- 分页展示，每页 20 条
- 页面顶部显示收藏总数

UI 需求：
- 使用现有的 ArticleCard 组件展示文章
- 分页使用数字分页器，显示总页数和当前页
- 加载中显示骨架屏
- 空状态显示"还没有收藏任何文章"提示

数据获取：
- 使用 useQuery，key 为 ['favorites', page]
- 请求 GET /api/users/me/favorites?page={page}&pageSize=20
```

AI 生成的前端代码质量不错，基本上拿来就能用。空状态的 UI 处理得很细致，加载骨架屏也用了正确的组件。这个页面没有返工，一次通过。

## 第六步：集成测试

前后端都实现完了，接下来需要验证整个功能是否正常工作。我用 curl 手动测试了所有接口，确认返回格式和数据都正确。然后在浏览器里走了一遍完整的用户流程：登录 → 打开文章 → 收藏 → 刷新页面看状态 → 去个人中心看收藏列表 → 取消收藏 → 确认状态变化。

测试过程中发现一个问题：取消收藏之后，文章详情页的收藏状态没有更新。这是因为 React Query 的缓存策略问题——我需要在取消收藏成功后，让 React Query 重新验证收藏状态的缓存。反馈给 AI：

```
取消收藏成功后，React Query 的收藏状态缓存没有更新，导致页面上显示的还是已收藏状态。
需要在取消收藏的 mutation 成功后，调用 queryClient.invalidateQueries(['favorite', articleId]) 刷新缓存。
```

AI 修复了这个问题。

## 第七步：代码审查和优化

功能跑通之后，我做了最后一轮代码审查，检查有没有遗漏的问题。

检查事项：

- 接口有没有做权限校验？（已做）
- 错误处理是否完善？（基本完善，但可以加上更详细的错误日志）
- SQL 有没有注入风险？（使用参数化查询，没问题）
- 前端有没有 XSS 风险？（文章标题和摘要使用了 React 的默认转义，没问题）
- 前后端接口字段命名是否一致？（检查了一遍，发现后端返回的字段是 `favorited` 但前端某个地方写的是 `isFavorited`，统一了）

最后做了一些小的优化：把收藏按钮的防抖时间从 3 秒改成了 1 秒（3 秒太长了，用户体验不好），给收藏接口加上了请求限流（防止用户快速点击）。

## 复盘

回顾整个开发过程，这个功能从拿到需求到测试通过，大概花了三个小时。如果纯手工写代码，同样的功能估计要一天以上。vibe coding 帮我节省了大量写"套路代码"的时间——数据库迁移、后端 CRUD 接口、前端状态管理，这些都是相对标准的实现，AI 写得又快又好。

但这不是一个"全程 AI 代写"的过程。我的投入主要在：

1. **写清晰的 Spec**——这一步占了整个过程中大约 20% 的时间，但决定了后续开发的效率
2. **审查 AI 的输出**——每个接口、每个组件生成之后，我都仔细 review 了一遍，发现并修正了若干问题
3. **处理边界情况**——乐观更新回滚、防抖、缓存失效，这些都是 AI 没有自动处理好的，需要人来指出来
4. **集成测试**——手动走了一遍完整流程，确保端到端没问题

vibe coding 不是"把需求丢给 AI 等结果"，而是一个"人机协作"的过程。AI 负责执行，人负责决策。什么时候让 AI 做，什么时候自己做，这个判断力决定了最终产出的质量。

## 流程总结

最后把这次开发的流程总结一下：

1. **写 Spec** —— 把需求写成清晰的文档，越详细越好
2. **基础设施先行** —— 数据库迁移先做，后面依赖它的代码才能写
3. **小步生成** —— 不要一次生成太多代码，逐个接口、逐个组件来做
4. **即时审查** —— AI 生成的代码立即 review，不要等到最后一起看
5. **反馈修正** —— 发现问题立刻反馈给 AI，让它修正后再继续
6. **端到端测试** —— 所有部分做完之后，走一遍完整的用户流程
7. **最后一轮审查** —— 从整体视角检查一遍有没有遗漏

这个流程适合大多数功能开发。不同项目、不同复杂度会有调整，但核心逻辑是相似的：Spec 先行，小步生成，即时验证，持续修正。
