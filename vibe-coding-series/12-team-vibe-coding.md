# Vibe Coding 实战（十二）：团队 vibe coding——规范、协作与知识积累

---

vibe coding 在个人开发者手里已经能发挥很大价值，但当团队开始使用的时候，复杂度会上升一个数量级。个人用 vibe coding，你就是自己的主人，想怎么迭代就怎么迭代；团队里用 vibe coding，你需要跟其他人协调规范、共享上下文、积累知识，否则每个人都在用自己的方式" vibe "，最后代码库会变成一锅粥。

这篇文章讲团队环境下做好 vibe coding 的方法。假设你的团队有 3 到 10 个人，大家在同一个代码库上工作，都有 AI 辅助编程的工具使用经验。

## 团队共享的 Rules 怎么管理

Rules 是个人项目中已经很有效的机制，团队里更是如此。但团队 Rules 面临几个个人 Rules 没有的挑战：谁来写、谁来改、怎么达成共识、怎么确保所有人都在用。

我建议的做法是：Rules 的所有权归团队，但维护权归个人。具体来说，基础 Rules（代码风格、接口规范、技术约束）由团队一起讨论确定，形成共识之后写入项目；具体模块的 Rules（比如某个复杂业务模块的特殊处理逻辑）由负责该模块的人维护，他们最了解这个模块的情况。

Rules 的变更流程应该跟代码变更一样，走 PR 流程。新增或修改一条 Rule，需要有人 review，确保这条 Rule 合理、没有歧义、不会对其他团队成员的工作造成干扰。这个流程听起来重，但实际上 Rule 的变更频率很低——项目规范一旦确定，不会频繁改动。

有一个常见的误区：觉得 Rules 越多越好。恰恰相反，Rules 越多，AI 读取的成本越高，实际执行效果越差。我的经验是，团队 Rules 控制在 20 条以内，只写最重要的、最容易出问题的、工具管不到的事情。代码风格统一靠 linter 配置，API 格式统一靠 OpenAPI schema，这些不需要写在 Rules 里。

Rules 的可发现性也很重要。团队 Rules 应该放在项目根目录的 `.cursor/rules/` 或者 `docs/rules.md` 里，新加入的成员第一件事就是读这个文件。Rules 写得再好，如果没人知道、没人读，就等于没有。

## vibe coding 的知识积累：好 Spec 的积累、好 prompt 的积累

vibe coding 模式下，最有价值的资产不是代码，而是知识：好的需求描述是怎么写的、好的 prompt 是怎么构造的、哪些需求描述带来了高质量的代码输出。

这些知识以前是"隐性"的，存在每个人的脑子里，用完就忘了。团队要高效运转，需要把这些隐性知识显性化，变成可积累、可复用的资产。

**好 Spec 的积累**。每个功能的 Spec（需求描述）是这个功能开发过程中最有价值的一份文档。Spec 写得好，AI 输出质量就高；Spec 写得粗糙，AI 输出就需要大量修正。团队应该建立 Spec 的积累机制：每个功能的 Spec 都保存下来，按功能类型分类组织，形成一个"Spec 知识库"。

后续开发类似功能的时候，可以先在知识库里找有没有可以参考的 Spec。比如要开发一个"点赞功能"，可以先找找有没有类似的"收藏功能"的 Spec，参考它的描述方式和细节颗粒度，然后根据新需求做调整。这比从零开始写 Spec 效率高得多，质量也更有保障。

**好 prompt 的积累**。除了 Spec，还有一些高频使用的 prompt 模式值得积累。比如"让 AI 解释一段代码的逻辑"、"让 AI 给一个函数补充测试用例"、"让 AI 审查某个模块的代码质量"——这些 prompt 的表述方式有优劣之分，一个好的 prompt 表述能让 AI 给出更好的结果。团队可以定期分享各自用下来效果最好的 prompt，形成团队的 prompt 最佳实践。

知识积累的关键是"用起来"。建立了知识库但没人去看，等于没建。我建议每个月团队例会上花 10 分钟分享一个"这月学到的好 prompt"或者"这个 Spec 写得特别好"——通过持续的分享，慢慢建立团队的知识积累文化。

## 团队协作中的上下文共享

个人 vibe coding 只需要管理自己的上下文，团队里还需要管理团队成员之间的上下文共享。

最常见的问题是：AI 生成的代码风格跟团队其他人写的代码不一致。这不是 AI 的问题，而是上下文不够的问题——AI 只知道你在 prompt 里给它了什么信息，不知道团队其他人的编码习惯。要解决这个问题，需要在 Rules 之外，给 AI 提供更多的上下文信息。

**分享代码审查中的 AI 使用记录**。团队成员在 Code Review 的时候发现 AI 经常犯的同类错误，应该把这个问题反馈给负责 Rules 维护的人，加一条新的规则。这个机制运行起来之后，团队里每个人的"踩坑经历"都能变成团队整体的"规避指南"。

**关键决策的文档化**。用 AI 开发某个功能的过程中做出的技术决策，应该记录下来。比如"这个功能用 Redis 做缓存而不是内存，因为部署环境是多实例的"——这个决策过程是 AI 不知道的，但后续维护的人需要知道。在代码里写清楚决策理由，或者在 docs/ 目录下维护一份技术决策记录，能帮助后续的 AI 协作。

**上下文传递的技巧**。当你把一个任务从 A 同事手里接过来继续做的时候，需要先了解 A 同事之前跟 AI 是怎么协作的、用过哪些 prompt、做过哪些决策。把这些信息作为你后续跟 AI 协作的上下文，而不是假设 AI 能自己从代码里推断出来。代码里看不到决策过程，但 AI 的输出质量高度依赖这个过程。

## 代码审查在 vibe coding 团队中的角色

代码审查在传统团队里是质量把关的环节，在 vibe coding 团队里又多了一个角色：AI 输出质量的把关。

AI 生成的代码通过 Code Review 发现问题，比在生产环境发现问题的代价低得多。但 Code Review 的方式也要调整。传统 Code Review 审查的是"代码是人写的，逻辑对不对、风格合不合规"；vibe coding 团队的 Code Review 还要审查"代码是 AI 写的，有没有 AI 容易犯的系统性错误"。

AI 生成代码有几个常见的系统性问题，Code Review 的时候要特别关注。

第一个是"过度工程"。AI 倾向于把代码写得很"完整"，加上各种可能用到的功能、异常处理、扩展点——即使这些在当前需求里完全不需要。审查的时候要判断：这段代码是为了解决什么问题？如果当前需求里没有这个场景，这个"扩展性设计"是必要的吗？

第二个是"上下文丢失"。AI 生成代码的时候只基于当前的 prompt 和读到的文件，可能忽略了项目里其他地方的约定和约束。审查的时候要对照 Rules 检查：这段代码是否符合团队的编码规范？变量命名跟团队风格一致吗？

第三个是"边界情况遗漏"。AI 倾向于处理"正常情况"，而忽略"异常情况"。审查的时候要刻意想想：空输入会怎样？并发场景会怎样？数据量大的情况会怎样？有没有可能导致系统不稳定的边界条件？

Code Review 的另一个价值是知识传递。Reviewer 看到被审查代码里的 AI 错误，可以把这个错误模式记录下来，加入 Rules 或者团队的常见问题文档。这样整个团队都会从单个错误里学到东西，错误不会再犯第二遍。

## 未来展望：vibe coding 成熟度模型

vibe coding 在团队里的应用深度不同，可以大致分成几个阶段。

**第一阶段：个人效率工具**。团队成员各自用 AI 辅助写代码，但没有统一的规范和协作机制。每个人都在用 AI，但用得参差不齐——有人效率很高，有人只是把 AI 当搜索引擎，有人甚至因为 AI 输出质量不稳定而放弃了 AI。这个阶段的特点是"有 AI 的使用"，但没有"AI 辅助开发的流程"。

**第二阶段：规范建立**。团队开始建立统一的 Rules，制定 prompt 模板，形成基本的协作流程。这个阶段团队整体的 AI 使用效率开始提升，但规范是静态的，需要人工维护和推动执行。

**第三阶段：流程整合**。AI 辅助开发跟团队的研发流程深度整合——Rules 自动同步、prompt 模板库可直接调用、Code Review 环节有 AI 错误模式的自动检测、好的 Spec 和 prompt 自动积累到知识库。这个阶段 AI 不是额外的工具，而是研发流程本身的组成部分。

**第四阶段：团队知识资产化**。经过长期积累，团队拥有丰富的知识库：各种场景的 Spec 模板、prompt 最佳实践、AI 错误模式记录、技术决策文档。新成员加入后能快速上手，因为知识库里已经有足够的上下文。这个阶段团队的 vibe coding 能力是可传承的，不依赖某几个"资深成员"的经验。

目前大多数团队在第一和第二阶段之间。走向第三和第四阶段需要时间和持续投入，但方向是清晰的。

## 团队 vibe coding 的几个原则

最后总结几个我认为最重要的原则。

**规范先行**。不要等到问题出现了再补规范。在团队开始使用 AI 辅助开发之前，先一起讨论确定编码规范、接口规范、Code Review 重点，形成共识。规范建立得越早，后面的协作成本越低。

**知识显性化**。每个人脑子里都有"怎么做更好"的经验，但这些经验如果只在自己脑子里，价值就只限于个人。主动把经验写出来——写成 Rules、写成文档、写成知识库条目——让团队所有人都能用上。

**持续复盘**。每隔一段时间，团队一起回顾一下最近用 AI 开发的项目：哪些地方用 AI 效果好、哪些地方效果不好、有什么可以改进的。这个复盘不需要很正式，半小时的小讨论就行。持续复盘能帮助团队快速迭代自己的 vibe coding 方法论。

**保持人的判断力**。AI 能帮你写代码，但不能替你做所有决策。架构设计、业务逻辑取舍、团队规范的制定——这些需要人来判断，AI 负责执行。不要因为 AI 能生成代码，就放弃了对"为什么要这样写"的思考。

vibe coding 在团队层面的成熟，跟个人层面的成长一样，需要时间和实践。但一旦团队建立了规范、积累了知识、形成了文化，AI 辅助开发的效率会远超个人层面的简单叠加——因为知识可以复用、错误不会重复犯、新人能快速上手。这是一条值得走的路。
